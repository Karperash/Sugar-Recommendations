DiaGuide CGM — соответствие разделам дипломной работы (2.4–2.6)
================================================================
Документ описывает, что уже реализовано в проекте, в привязке к пунктам
2.4 (алгоритм обработки и анализа данных), 2.5 (правила рекомендаций),
2.6 (средства и технологии). Текст с меткой [НЕ РЕАЛИЗОВАНО — перспектива]
относится к возможному развитию, в текущем коде этого нет.


--------------------------------------------------------------------------------
2.4. Алгоритм обработки и анализа данных
--------------------------------------------------------------------------------

РЕАЛИЗОВАНО:

• Ввод и нормализация на уровне прототипа
  — Ручной ввод измерений глюкозы (с трендом, опционально меткой приёма пищи).
  — Импорт потоков записей из CSV и JSON (парсеры CgmCsvParser, CgmJsonParser);
    данные приводятся к модели CgmRecord и сохраняются в локальное хранилище.
  — Генерация синтетических сценариев для демонстрации и тестов
    (SyntheticCgmDataFactory, сценарии: стабильный день, повторные гипо/гипер,
    резкие колебания, ночные эпизоды и др.).

• Хранение и выборка по времени
  — Room: записи CGM, рекомендации, события; выборка по интервалу дат
    для анализа «за сутки» и отображения на экранах.

• Ядро анализа — класс CgmAnalysisEngine (domain-слой, правила на Kotlin)
  — Фильтрация записей за выбранные сутки и часовой пояс (LocalDate, ZoneId).
  — Построение дневной сводки DailySummary: минимум, максимум, среднее,
    доля времени в целевом диапазоне, счётчики ниже/выше цели (пороги из
    UserSettings: target, warning, critical).
  — Детекция паттернов (каждый — DetectedPattern с типом, пояснением,
    серьёзностью, временем):
      - критически низкие / высокие значения относительно порогов;
      - выход за целевой диапазон (ниже target low / выше target high);
      - быстрый рост / падение между соседними точками (нормировка на 15 мин,
        пороги rapidRise / rapidFall из настроек);
      - длительное нахождение вне целевого диапазона (≥ prolongedOutOfRangeMinutes);
      - повторяющиеся выходы за диапазон в скользящем окне patternWindowHours;
      - «утренние» высокие значения (окно часов 6–10, ≥ 2 эпизода);
      - ночные низкие значения (0–5 ч., ≥ 2 эпизода);
      - возможный постпрандиальный скачок (метка meal + окно до 120 мин).
  — Итог анализа: структура AnalysisResult (сводка + список паттернов);
    дубликаты по типу паттерна убираются (distinctBy type).

• Связка с остальной логикой приложения
  — Use cases: AnalyzeGlucoseTrendsUseCase, GetDailySummaryUseCase (реактивно
    через Flow + combine), RefreshInsightsUseCase — загрузка настроек и записей
    за день, вызов анализа, передача паттернов дальше в модуль рекомендаций
    и сохранение списка рекомендаций в БД.

[НЕ РЕАЛИЗОВАНО — перспектива]

• Математические / ML-модели прогноза глюкозы (ARIMA, LSTM, физиологические
  симуляторы и т.п.) — сейчас только эвристики и пороговая логика.
• Автоматическая нормализация единиц и форматов при смешанном импорте
  (в README указано упрощение: единицы должны соответствовать выбранной в приложении).
• Интеграция с реальными CGM по BLE/облаку — интерфейс заглушки есть,
  передача «живых» данных с устройства не реализована.
• Калибровка сенсора, фильтрация шума Калмана/медианные сглаживания — нет.
• Оценка качества данных (пропуски, выбросы по статистике) — минимально.


--------------------------------------------------------------------------------
2.5. Правила формирования рекомендаций
--------------------------------------------------------------------------------

РЕАЛИЗОВАНО:

• Модель «паттерн → рекомендация» — класс RecommendationEngine
  — На вход: список DetectedPattern + UserSettings; на выход: List<Recommendation>.
  — Детерминированное отображение: when (pattern.type) с заготовленными
    текстами title, shortExplanation, severity, actionButtonLabel (подсказка
    действия в UI: открыть график, историю, добавить заметку и т.д.).
  — Уникальный id рекомендации (UUID), привязка к relatedDetectedPattern.

• Уровни серьёзности (RecommendationSeverity) согласованы с анализом:
  URGENT для критических порогов, WARNING для большинства отклонений и
  повторяющихся сценариев, INFORMATIONAL для утренних высоких и постпрандиальных
  всплесков (как в логике CgmAnalysisEngine).

• Сохранение и история
  — RefreshInsightsUseCase заменяет набор рекомендаций за расчётный день в Room;
  — экран истории рекомендаций и дашборд показывают сформированные записи.

• Граница применения (важно для текста диплома)
  — Рекомендации носят информационно-образовательный характер; расчёт доз
    инсулина и диагностика в приложении явно не выполняются (см. README,
    дисклеймеры в UI).

[НЕ РЕАЛИЗОВАНО — перспектива]

• Персонализация текстов под профиль (возраст, тип диабета) в правилах — поля
  профиля есть, но шаблоны рекомендаций от них не зависят.
• Локализация текстов рекомендаций внутри domain (сейчас строки на английском
  в RecommendationEngine; интерфейс приложения может переключаться RU/EN).
• Веса правил, приоритеты, объединение нескольких паттернов в одну
  «комплексную» рекомендацию — сейчас по сути «один паттерн — одна карточка»
  (с дедупликацией паттернов по типу на этапе анализа).
• Обратная связь пользователя («полезно / не полезно») для дообучения или
  коррекции правил — нет.
• Формальная онтология/стандарты клинических guideline (например, автоматический
  вывод по ADA/EASD) — не подключались; правила заданы вручную в коде.


--------------------------------------------------------------------------------
2.6. Средства и технологии реализации
--------------------------------------------------------------------------------

РЕАЛИЗОВАНО:

• Язык и среда: Kotlin, JDK 17, Android Gradle Plugin, сборка через Gradle
  (версии см. libs.versions.toml / app/build.gradle.kts: minSdk 26,
  compileSdk / targetSdk 36).

• UI: Jetpack Compose, Material 3, Navigation Compose; кастомная отрисовка
  графика CGM (Canvas), тема и типографика.

• Архитектура: MVVM, Clean Architecture по пакетам (presentation / domain / data),
  контракты репозиториев в domain, реализации в data.

• DI: Hilt (kapt), модули привязки Room/парсеров/движков.

• Локальное хранение: Room (DAO, миграции с fallback destructive в прототипе),
  DataStore Preferences (профиль, пороги, язык приложения, онбординг).

• Асинхронность: Kotlin Coroutines, Flow; связка UI — ViewModel + StateFlow/State.

• Фоновые задачи: WorkManager — периодические напоминания (ReminderWorker),
  канал уведомлений.

• Локализация UI: strings.xml (английский по умолчанию, values-ru), выбор языка
  в настройках через AppCompat per-app locales; MainActivity на AppCompatActivity.

• Версионирование: versionName / versionCode в Gradle; журнал VERSION_HISTORY.txt.

• Тестирование: модульные тесты (JUnit, Truth, Turbine) для парсеров,
  CgmAnalysisEngine, RecommendationEngine, части ViewModel (см. app/src/test).

[НЕ РЕАЛИЗОВАНО — перспектива]

• Полноценный CI/CD, автоматические UI-тесты на все сценарии — только точечные
  androidTest-зависимости, без описанного конвейера.
• Модульное разбиение на несколько Gradle-модулей (data/domain отдельно) —
  сознательно один модуль app для упрощения дипломного прототипа.
• Серверный backend, синхронизация между устройствами, аккаунты — нет.
• KSP вместо kapt — в проекте по-прежнему kapt для Room и Hilt.
• Сквозная сквозная телеметрия / аналитика — нет.
